# Домашнее задание №4 — Notification Platform

## Задания 1–7: Требования и архитектурное мышление

---

## 1. Функциональные требования

### FR1. Доставка транзакционных уведомлений

**Как** пользователь онлайн-банка,
**я хочу** получать уведомление о каждой финансовой операции (списание, перевод, пополнение),
**чтобы** я мог контролировать движение средств и вовремя заметить подозрительную активность.

*Связь с бизнес-целью:* гарантированная доставка критичных уведомлений; снижение жалоб на 30%.

### FR2. Управление настройками уведомлений

**Как** пользователь,
**я хочу** выбирать предпочтительный канал доставки (push, SMS, email) и отключать сервисные и маркетинговые
уведомления,
**чтобы** получать только важную для меня информацию удобным способом.

(транзакционные уведомления (списания, переводы, подтверждения операций) нельзя отключить — они связаны с безопасностью.
Пользователь может выбрать канал, но не отключить их полностью)

*Связь с бизнес-целью:* снижение жалоб на уведомления на 30%.

### FR3. Автоматический failover между каналами

**Как** система,
**я должна** автоматически переключиться на резервный канал доставки, если основной канал недоступен или не подтвердил
доставку в отведённое время,
**чтобы** критичное уведомление было гарантированно доставлено.

*Связь с бизнес-целью:* гарантированная доставка критичных уведомлений.

### FR4. Массовая рассылка маркетинговых уведомлений

**Как** маркетолог,
**я хочу** отправлять маркетинговые кампании (акции, кэшбэк-предложения) аудитории до 1 млн пользователей,
**чтобы** повысить вовлечённость и retention.

*Связь с бизнес-целью:* увеличение retention на 15%.

### FR5. Дедупликация уведомлений

**Как** пользователь,
**я не хочу** получать одно и то же уведомление дважды (в том числе через разные каналы),
**чтобы** не путаться и не терять доверие к системе.

*Связь с бизнес-целью:* снижение жалоб на уведомления на 30%.

### FR6. История уведомлений

**Как** пользователь,
**я хочу** видеть историю полученных уведомлений в приложении с информацией о статусе доставки,
**чтобы** вернуться к любому уведомлению в удобное время.

*Связь с бизнес-целью:* увеличение retention на 15% (пользователь не теряет информацию).

### FR7. Мониторинг доставки для операционной команды

**Как** оператор службы мониторинга,
**я хочу** в реальном времени видеть дашборд со статусами доставки, success rate и количеством failover-ов по каждому
каналу,
**чтобы** оперативно реагировать на деградацию канала или провайдера.

*Связь с бизнес-целью:* гарантированная доставка критичных уведомлений; централизованный контроль.

## 3. Архитектурно значимые требования (ASR)

### ASR1: Низкая задержка доставки транзакционных уведомлений

**Связанные требования:**

- FR1 (доставка транзакционных уведомлений)
- NFR1 (latency ≤ 3 сек, p95)

**Почему влияет на архитектуру:**
Требует минимизации количества промежуточных компонентов (hop-ов) на пути транзакционного уведомления. Диктует
необходимость отдельных приоритетных очередей для критичных уведомлений, чтобы маркетинговые и сервисные не создавали
backpressure. Влияет на модель взаимодействия компонентов — предпочтение асинхронных неблокирующих вызовов с
гарантированным временем обработки.

**Приоритет:** Критический

---

### ASR2: Гарантированная доставка с кросс-канальным failover

**Связанные требования:**

- FR3 (автоматический failover)
- FR5 (дедупликация)
- NFR4 (delivery rate ≥ 99.9%)

**Почему влияет на архитектуру:**
Требует персистентного хранения состояния каждого уведомления (state machine), чтобы система знала, на какой стадии
находится доставка. Необходимы idempotency-ключи для предотвращения дублей при failover. Требует механизма таймаутов и
callback-ов от провайдеров каналов. Определяет паттерн координации компонентов (оркестрация vs хореография).

**Приоритет:** Критический

---

### ASR3: Высокая пропускная способность

**Связанные требования:**

- FR4 (массовые рассылки до 1M)
- NFR2 (availability 99.95%)
- NFR3 (30M уведомлений/день, пик 1 750 rps)

**Почему влияет на архитектуру:**
Требует горизонтального масштабирования всех компонентов системы. Определяет выбор брокера сообщений (должен
поддерживать партиционирование и высокий throughput). Необходим rate limiting к внешним провайдерам, чтобы не превысить
их лимиты при пиковых нагрузках. Массовые рассылки должны обрабатываться отдельным pipeline-ом, чтобы не влиять на
транзакционные уведомления.

**Приоритет:** Высокий

---

### ASR4: Наблюдаемость процесса доставки

**Связанные требования:**

- FR6 (история уведомлений)
- FR7 (мониторинг в реальном времени)
- NFR5 (полный трейсинг, метрики по каналам)

**Почему влияет на архитектуру:**
Каждый компонент должен эмитить структурированные события (structured logging + distributed tracing). Необходим
централизованный сбор метрик (push-модель или pull-модель). Требует хранилища для аудит-лога уведомлений. Влияет на
формат межсервисного взаимодействия — каждый запрос должен нести trace ID для сквозного трейсинга.

**Приоритет:** Высокий

## 4. Ключевые архитектурные вопросы

### Q1: Как обеспечить идемпотентность при failover?

**Порождается:** ASR2 (гарантированная доставка + дедупликация)

**Суть проблемы:** При failover между каналами существует окно, когда основной канал (например, push) мог доставить
сообщение с задержкой, а система уже инициировала отправку через SMS. В результате пользователь получает дубликат.

**Почему важен:** Дублирование транзакционных уведомлений (например, двойное уведомление о списании) напрямую подрывает
доверие пользователя и может спровоцировать обращения в поддержку.

---

### Q2: Как приоритизировать транзакционные уведомления при конкуренции с маркетинговыми?

**Порождается:** ASR1 (низкая задержка) + ASR3 (пропускная способность)

**Суть проблемы:** Массовая маркетинговая рассылка на 1M пользователей генерирует ~280 rps дополнительной нагрузки. Если
транзакционные и маркетинговые уведомления используют общую инфраструктуру, массовая рассылка может увеличить latency
транзакционных уведомлений выше SLA.

**Почему важен:** Нарушение SLA транзакционных уведомлений — это и деградация UX, и потенциальные проблемы с
регулятором (своевременное уведомление о списаниях).

---

### Q3: Какой паттерн координации использовать для failover — оркестрация или хореография?

**Порождается:** ASR2 (гарантированная доставка) + ASR1 (низкая задержка)

**Суть проблемы:** Оркестрация (центральный сервис управляет цепочкой failover) даёт явный контроль и observability, но
создаёт потенциальную точку отказа. Хореография (события через брокер) устраняет SPOF, но усложняет мониторинг и дебаг.

**Почему важен:** Этот выбор определяет core-архитектуру подсистемы доставки и влияет на сложность разработки,
развёртывания, отладки и масштабирования.

---

### Q4: Как обеспечить устойчивость при недоступности внешних провайдеров?

**Порождается:** ASR2 (гарантированная доставка) + NFR2 (availability 99.95%)

**Суть проблемы:** Внешние провайдеры (SMS-шлюзы, email-сервисы, FCM/APNs) могут быть недоступны от секунд до часов.
Система должна продолжать работать: не терять уведомления, корректно переключаться между каналами и провайдерами.

**Почему важен:** Без стратегии устойчивости (circuit breaker, retry с backoff, fallback-провайдер) один упавший
SMS-шлюз может привести к каскадному отказу всей цепочки доставки.

## 5. Архитектурные последствия ASR

### ASR1: Низкая задержка транзакционных уведомлений

**Последствия:**

- Отдельные приоритетные очереди/топики для транзакционных уведомлений (физическая изоляция от маркетинговых)
- Асинхронная обработка с минимумом hop-ов (API → брокер → worker → провайдер)
- Pre-computed routing: канал доставки определяется заранее на основе кэшированных настроек пользователя (Redis), а не
  запрашивается из БД в момент обработки
- Dedicated consumer group для транзакционных уведомлений с гарантированной ёмкостью

### ASR2: Гарантированная доставка с failover

**Последствия:**

- Персистентное хранение состояния уведомления (state machine в БД)
- Idempotency key для каждого уведомления (Redis с TTL)
- Механизм таймаутов: если канал не подтвердил доставку за N секунд — failover
- Delivery receipt tracking: callback/webhook от провайдеров или polling статуса
- Уведомление сначала сохраняется в БД, и только потом публикуется в брокер — так мы не потеряем его даже при сбое (
  at-least-once)

### ASR3: Высокая пропускная способность

**Последствия:**

- Горизонтальное масштабирование всех stateless-компонентов
- Партиционирование очередей по user_id (гарантия порядка для одного пользователя)
- Ограничение частоты запросов к внешним провайдерам (rate limiting), чтобы не превысить их лимиты
- Отдельный pipeline для массовых рассылок с throttling
- Batch-обработка маркетинговых уведомлений

### ASR4: Наблюдаемость

**Последствия:**

- Distributed tracing (trace_id пробрасывается через все компоненты)
- Structured logging в JSON-формате
- Метрики: Prometheus-совместимые counter/histogram для каждого этапа
- Аудит-лог в append-only хранилище (для compliance)
- Real-time дашборд с алертами на деградацию SLA

## 6. Архитектурные решения, которые НЕ подходят

### Антипаттерн 1: Синхронная доставка уведомлений

**Описание:** Сервис-инициатор (например, сервис переводов) синхронно вызывает API внешнего провайдера (SMS-шлюз, FCM) и
ждёт ответа в рамках HTTP-запроса пользователя.

**Какой ASR нарушается:** ASR1 (низкая задержка), ASR3 (масштабируемость)

**Почему:**

- Время ответа внешнего провайдера непредсказуемо (от 100ms до 30+ сек при деградации). Синхронный вызов блокирует поток
  сервиса-инициатора, что при пиковой нагрузке приведёт к исчерпанию пула потоков и каскадному отказу.
- Невозможно реализовать failover: если push не доставлен, пользователь уже получил ответ от API, и повторная отправка
  через SMS требует отдельного асинхронного процесса.
- Не масштабируется для массовых рассылок: 1M синхронных вызовов к SMS-шлюзу = часы блокирующей работы.

---

### Антипаттерн 2: Единая очередь для всех типов уведомлений

**Описание:** Все уведомления (транзакционные, сервисные, маркетинговые) помещаются в одну очередь/один топик и
обрабатываются одним пулом consumer-ов по порядку.

**Какой ASR нарушается:** ASR1 (низкая задержка транзакционных)

**Почему:**

- Массовая маркетинговая рассылка на 1M пользователей моментально заполняет очередь. Транзакционное уведомление о
  списании попадает в хвост и ждёт обработки миллионов маркетинговых сообщений.
- При пиковой нагрузке (1 750 rps) consumer lag единой очереди может достигать минут, что прямо нарушает SLA в 3 секунды
  для транзакционных уведомлений.
- Нет возможности независимо масштабировать обработку разных типов уведомлений.

## 7. Неопределённости и архитектурные риски

### Неопределённость 1: Реальная надёжность push-уведомлений

**Что неизвестно:** Подтверждение от FCM/APNs (HTTP 200) означает, что сообщение принято платформой, но **не гарантирует
** доставку на устройство пользователя. Устройство может быть оффлайн, приложение удалено, уведомления отключены на
уровне ОС.

**Влияние:** Если мы считаем push "доставленным" по HTTP 200, реальный delivery rate может быть значительно ниже 99.9%.
Это критично для транзакционных уведомлений.

**Как проверить:**

- Собрать статистику реальной доставки push-уведомлений через FCM/APNs (например, через delivery receipts или
  отслеживание открытий)
- A/B тест: для выборки пользователей сравнить push-only vs push+SMS fallback по метрике "пользователь прочитал
  уведомление"
- По результатам определить оптимальный таймаут ожидания push-confirmation перед failover на SMS

### Неопределённость 2: SLA и ценообразование внешних SMS/email провайдеров

**Что неизвестно:** Реальный SLA конкретных SMS-шлюзов и email-провайдеров (SMSC, МТС Exolve, Sendgrid и т.д.) при нашем
объёме трафика. Заявленный SLA может отличаться от фактического. Ценообразование может меняться при росте объёмов.

**Влияние:** Если единственный SMS-провайдер недоступен часами, все failover-ы на SMS канал будут провалены.
Непредвиденный рост стоимости SMS при высоком failover rate может многократно увеличить операционные расходы.

**Как проверить:**

- Провести нагрузочное тестирование с 2-3 SMS-провайдерами, замерить реальные latency, success rate, throughput limits
- Запросить у провайдеров SLA-гарантии с компенсациями (SLA credits)
- Реализовать стратегию с несколькими провайдерами, чтобы можно было переключаться между ними при сбоях
- Внедрить мониторинг стоимости SMS в реальном времени с алертами при превышении бюджета
